# Politica de Changelog e Release Notes

## Objetivo
Padronizar como mudanças relevantes do produto sao comunicadas ao time e aos consumidores internos/externos.

## Deve Entrar em Release Notes
- mudanca de contrato de API
- novo caso de uso exposto
- mudanca em politicas institucionais suportadas
- mudanca em boletins, relatorios ou dashboard
- mudanca em autenticacao, autorizacao ou multi-tenancy
- mudanca relevante de observabilidade e operacao

## Nao Precisa Entrar Sempre
- refactor interno sem impacto funcional
- ajuste de teste sem alterar comportamento
- limpeza de codigo sem alterar contrato

## Estrutura Recomendada
- o que mudou
- impacto funcional
- impacto tecnico/operacional
- migracoes ou cuidados necessarios
